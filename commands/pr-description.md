PR 디스크립션을 작성한다. 아래 구조를 따른다.

## 구조
크게 세 섹션으로 구성한다.

### 1. Work Description
- 번호가 매겨진 소제목들로 의사결정 단위를 나눈다(예: "1. 계층형 아키텍처", "2. 단방향 데이터 흐름", "3. Room 기반 영속화", ...).
- 각 소제목은 "한 가지 설계 결정 / 한 가지 기능 / 한 가지 버그 수정"을 담는다. 커밋을 나열하지 말고 의도 단위로 묶는다.
- 각 항목은 bullet 2~4개로 구성한다. "무엇을 했다 / 왜 그렇게 했다 / 어떤 트레이드오프가 있었다"가 한 묶음.
- 변경된 클래스·함수명을 그대로 노출한다(`backtick`). 단순 "추가했습니다"로 끝내지 않고 어떤 규칙·불변식·시그니처를 들여왔는지 한 문장 더 적는다.
- 버그 수정도 별도 항목으로 둔다. 표면 증상보다 "왜 그 버그가 났는지" 원인을 한 줄로 적는다.

### 2. Screenshot
- "상품 목록 / 상품 상세 / 장바구니"처럼 **화면 단위로 표 형태**로 정렬한다.
- 영상이 있으면 `<video src="...">`, 없으면 `(영상/스크린샷 첨부 예정)` placeholder로 둔다.

### 3. To Reviewers
- 인사 한 줄로 시작. 이모지 사용 가능(이 섹션만 예외).
- 리뷰어에게 의견을 구하고 싶은 지점을 bullet 2~4개로. 각 bullet은:
  1. 어떤 결정을 했는지(한 문장)
  2. 그 결정의 근거(한 문장)
  3. 리뷰어에게 묻고 싶은 구체적 질문(한 문장)
- README 링크 등 부수 정보는 첫 번째 bullet 위치에.

## 톤과 스타일
- 한국어로 작성. 본문은 격식체("~했습니다").
- "Work Description" 본문에서는 이모지 사용하지 않음. "To Reviewers" 섹션에서만 가벼운 이모지(🙇‍♂️ 🤔 🙏) 허용.
- 각 항목은 **결정의 동기**를 드러낸다. "X를 도입했다"가 아니라 "Y라는 문제를 해결하기 위해 X를 선택했다"의 형태.
- 추상화 거부 사유, 라이브러리 미도입 사유 등 "하지 않은 것에 대한 설명"도 적극적으로 포함한다. 미션 수준에서 과도한 추상화를 거부했다는 식의 멘트는 가치가 있다.
- "기본값을 호출 측이 재정의할 수 있도록 열어두었다"처럼 미래 확장성에 대한 짧은 코멘트를 자연스럽게 끼워넣는다.

## 작성 절차
1. `git log <base>..<head> --oneline`과 `git diff <base>..<head> --stat`로 실제 변경 내역을 먼저 확인한다.
2. 커밋들을 의도 단위로 묶어 Work Description의 소제목들을 만든다.

## 참고 PR이 주어졌을 때 (중요)
사용자가 다른 사람의 PR 디스크립션을 참고로 보여주는 경우, **빌려오는 것은 뼈대뿐**이다.

빌려와도 되는 것 (뼈대):
- 큰 섹션 구성 (Work Description / Screenshot / To Reviewers)
- 번호 매긴 소제목 + bullet 2~4개의 형식
- Screenshot 표 레이아웃
- To Reviewers의 "결정 + 근거 + 질문" 3-bullet 패턴

절대 가져오지 말 것:
- 참고 PR의 **소제목 라벨**(예: "계층형 아키텍처", "원시값 포장 Value Object 적용") — 그 라벨은 그 사람의 작업 분류이지 내 작업이 아니다.
- 참고 PR의 **기술 선택**(예: retain, StateHolder, value class) — 내 프로젝트가 실제로 한 선택이 아니면 쓰지 않는다.
- 참고 PR의 **고유 문장 패턴**("도메인 규칙이 호출 경로와 무관하게 항상 보장됩니다" 같은 그 사람의 말투) — 베끼지 말고 다시 쓴다.

작성을 마친 뒤 자가 점검: 내 PR 디스크립션의 소제목/문장을 참고 PR 옆에 두고 비교했을 때 "거의 똑같네"라는 인상이 들면 실패. 형식이 같아야지 내용이 같으면 안 된다.
