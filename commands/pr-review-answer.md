PR 리뷰 코멘트에 대해 "이건 무슨 뜻이야?", "어떻게 고쳐야 해?" 같은 질문을 받았을 때 아래 구조를 따라 답한다.

## 구조
1. **원래 질문 복원** — 대화가 길어져 문맥이 흐려졌다면, 리뷰어의 원 질문을 한 문장으로 다시 적어 문맥을 재수립한다.
2. **핵심 개념 먼저 정의** — 답에 쓸 용어(예: "컴파일타임 의존성 vs 런타임 의존성", "일급 컬렉션", "디미터 법칙")를 bullet로 짧게 정의하고, 그 개념이 왜 이 질문과 관련 있는지 한 줄로 연결한다.
3. **원칙/명제 제시** — 객사오·오브젝트·DDD 등에서 나오는 관련 원칙을 한 문장으로 인용하듯 제시한다. 예: *"컴파일타임 의존성과 런타임 의존성은 서로 다를수록 유연한 설계다."*
4. **같은 렌즈로 여러 대안 비교** — A, B, C 같은 라벨을 달고, **모든 대안을 동일한 소제목 구조로** 분석한다. 예: 각 대안마다 "컴파일타임 의존성 / 런타임 의존성 / 두 의존성의 관계"를 일정한 순서로 짚는다. 일관된 포맷이 비교 가능성을 만든다.
5. **한 걸음 더 (심화)** — 바로 적용하지 않더라도 같은 원리를 더 밀고 나가면 어떻게 되는지 보여준다. 예: 추상화, 인터페이스화, 다형성 도입. "지금 당장 할 필요는 없지만 원리는 같다"고 선을 긋는다.
6. **원래 코드의 진짜 문제 진단** — 표면적 증상이 아니라 **리뷰어가 돌려서 묻는 근본 이유**를 한 문단으로 날카롭게 번역한다. "리뷰어는 사실 ~을 묻고 있다" 형식.
7. **리뷰어 코멘트 재번역** — 리뷰어의 짧은 코멘트가 사실 어떤 큰 질문을 품고 있었는지 `>` 인용으로 다시 써준다.
8. **정리 (스펙트럼/스케일)** — 방안들을 유연성·결합도 등 **하나의 축 위에 배치**해 스펙트럼으로 요약한다. "왼쪽=경직, 오른쪽=유연" 같은 방식.
9. **현재 사용자의 방향 평가** — 사용자가 지금 가고 있는 리팩토링 방향을 그 스펙트럼 안에서 위치 지정하고, "이 선택이 스펙트럼상 어디에 해당하며 무엇까지 같이 해결되는지" 짚어준다. 마지막에 리뷰어의 의도와 연결해 마무리한다.

## 톤과 스타일
- 코드 블록을 각 대안마다 짧게 넣어 **시각적 비교**가 되게 한다.
- Bullet + 굵은 글씨로 "컴파일타임 의존성 / 런타임 의존성 / 관계" 같은 고정 항목을 반복해 리듬을 만든다.
- "이게 나쁜 건 X 때문이 아니라 Y 때문이다" 같은 **오해 교정 문장**을 한 번은 넣는다.
- 마지막 단락은 리뷰어의 의도를 해석하며 "작은 질문으로 큰 설계 원칙을 건드렸다"는 식의 메타 코멘트로 마무리한다.
- 이모지는 사용하지 않는다. 한국어로 작성한다.
