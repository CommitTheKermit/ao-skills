# release-commit

배포(또는 버전 릴리스) 직후 남기는 커밋을 **일정한 포맷**으로 작성하고, 본문은 **직전 배포 이후의 git 내역에서 자동으로 뽑아** 채운다. 한두 줄 요약만 남겨 "이번 배포에 무엇이 들어갔는지" 추적이 안 되는 문제를 막는 것이 목적이다.

## 고정 포맷

```
chore(release): v<X.Y.Z> 배포

대상: <배포 대상> / <A.B.C -> X.Y.Z> (<major|minor|patch>)

- <type(scope): 요약>   # 직전 배포 이후 커밋들을 한 줄씩
- ...

<선택: 배포 URL 등 프로젝트 고유 정보>
```

- 제목은 항상 `chore(release): v<X.Y.Z> 배포`. 버전엔 `v` 접두를 고정으로 붙이고, 대상/상세는 본문에 둔다. `버전 X 로 올리고 배포`·`(hosting)` 같은 제각각 변형은 쓰지 않는다.
- 첫 본문 줄(`대상: ...`)에 **무엇을 어디에** 배포했는지(예: functions + hosting, npm publish 등)와 **버전 증감**(`A.B.C -> X.Y.Z`)·증가 단위(major/minor/patch)를 적는다.
- 빈 줄 뒤에 변경 목록을 둔다.
- 배포 URL·콘솔 링크 등 프로젝트 고유 정보는 맨 아래에 선택적으로 덧붙인다.

## 본문 변경 목록 만드는 법

직전 release 이후 쌓인 커밋을 git 에서 그대로 뽑아 옮긴다. 임의로 요약하거나 누락하지 말 것.

1. **직전 release 기준점 찾기**
   ```bash
   git log --grep='chore(release)' --format=%H -n 1
   ```
   - 결과가 없으면(첫 release) 마지막 **버전 변경 커밋**을 기준으로 삼는다(예: `git log -S '"version"' -- <버전파일> | head`), 그것도 없으면 최근 태그(`git describe --tags --abbrev=0`).
   - 주의: `git log -S '"version": "<직전버전>"'` 는 **새 버전으로 올리는 커밋까지** 잡아 범위가 비는 함정이 있다. bump 커밋 자체를 해시로 특정해 기준으로 쓴다.

2. **범위 커밋 수집**
   ```bash
   git log <직전 release 또는 기준 커밋>..HEAD --oneline
   ```
   - release 커밋을 만들기 **전에** 실행한다(버전 bump 커밋이 HEAD 이면 `..HEAD^` 로 그 직전까지).

3. **목록으로 정리**
   - 각 커밋을 `- type(scope): 요약` 한 줄로 옮긴다. 원문 요약을 살리되 사람이 읽기 좋게 다듬는다.
   - release/단순 버전 bump 커밋은 목록에서 제외한다.
   - 항목이 많으면 성격별로 묶어도 되지만(예: 렌더링/스트리밍/세션), 개별 커밋이 사라지지 않게 한다.

## 적용 절차

1. 버전을 올리고(프로젝트의 버전 단일 출처 파일) 빌드/배포를 끝낸다.
2. 위 방법으로 직전 release 이후 변경 목록을 수집한다.
3. 고정 포맷으로 release 커밋을 작성한다(서명 트레일러 없음, 한국어 본문).
4. 직전 배포가 release 커밋 없이 버전만 올라간 경우, 그 누락분까지 이번 목록에 포함되니 기준점을 정확히 잡는다.

## 이미 남긴 빈약한 release 커밋 보강

직전 release 커밋이 한두 줄로 끝나 있고 **아직 push 되지 않았다면**(`git branch -r --contains <해시>` 가 비어 있음) `git commit --amend` 로 본문만 교체한다. amend 는 staged 변경만 반영하므로, 무관한 작업물은 stage 하지 않은 채 메시지만 바꾼다. 이미 push 됐다면 history rewrite 가 되므로 사용자에게 force push 여부를 먼저 확인한다.
