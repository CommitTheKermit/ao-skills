# STT 검수본·요약본 작성 스킬

STT(Speech-to-Text) 도구(다글로(Daglo), 클로바, 맥북 녹음 등)에서 추출된 녹취 텍스트를 **검수본**과 **요약본** 두 개의 마크다운 파일로 변환하는 스킬입니다.

STT 결과물의 전형적인 약점 - 기술 용어 오인식, 인명·고유명사 왜곡, 끊긴 문장, 화자 추임새 노이즈, 화자 분리 오류 - 을 대화 맥락 기반으로 보정하고, 별도로 핵심 내용을 구조화된 요약본으로 추출합니다.

## 작업 흐름

### 1단계 - STT 파일 읽기

사용자가 파일을 업로드했다면 그 파일을 먼저 읽습니다. 텍스트로 붙여넣었다면 그대로 사용합니다.

대규모 파일이면 전체를 통째로 읽지 말고 앞·중간·끝을 분할해 훑어, 대화 주제·등장 인물·세션 길이를 먼저 파악합니다.

### 2단계 - 사용자에게 맥락 확인 (반드시 수행)

작성 전에 사용자에게 다음을 적극적으로 물어 맞춤 결과를 만듭니다. **추측만으로 진행하지 마세요.** 한 번에 하나씩 또는 ask_user_input_v0로 묶어서 물어볼 수 있습니다.

- **세션 맥락**: 어떤 주제·성격의 모임인가? (스터디, 회의, 강의, 합평, 인터뷰 등)
- **참고 메모**: 사용자가 회의 중 적어둔 메모, 발표자 자료, 관련 문서가 있는가? 있다면 첨부 요청
- **화자 정보**: 화자 수와 화자 매핑. 사용자가 따로 지시하지 않으면 **A, B, C, …로 익명 표기**가 기본
- **연속 회차 여부**: 이전 회차 검수본이 존재하는가? 통합본까지 만들 것인가? (회차가 누적되면 화자 표기를 이전 회차와 일치시켜야 함)

맥락 확인을 건너뛰면 화자 표기 오류·용어 오인식·구조 부적합이 발생해 결과물을 다시 만들어야 하는 경우가 잦습니다. 그래서 이 단계가 결과 품질의 절반을 좌우합니다.

### 3단계 - 검수본 작성

세부 규칙은 아래 **「검수본 작성 규칙」** 절을 따릅니다. 핵심은 **STT 오류 보정 + 화자 표기 정리 + 문장 다듬기 + 끝에 보정 사항 주석**입니다.

### 4단계 - 요약본 작성

세부 규칙은 아래 **「요약본 작성 규칙」** 절을 따릅니다. 검수본의 발언을 토픽별로 재구성하고, 사용자가 제공한 참고 메모의 인사이트도 반영합니다.

### 5단계 - 파일 출력 및 공유

두 파일을 `/mnt/user-data/outputs/` 아래에 저장하고 `present_files`로 공유합니다. 파일명은 `검수본_<세션명>.md`, `요약본_<세션명>.md` 형식을 권장합니다.

연속 회차에서 통합본을 요청받았다면 `검수본_<세션명>_통합.md`, `요약본_<세션명>_통합.md`도 함께 만듭니다. 통합 시 세부 규칙은 **「통합본 작성 규칙」** 절을 따릅니다.

## 검수본 작성 규칙

### 머리말 형식

문서 최상단에 다음 요소를 포함합니다:

```markdown
# <세션명> 검수본

> STT 결과를 <대화 맥락>에 맞춰 음성 인식 오류와 끊긴 문장을 다듬은 검수본입니다.
> 화자는 <화자 표기 정책>로 구분했습니다. <필요시 화자 매핑 설명>

---
```

### 본문 형식 - 발언 단위

각 발언은 **타임스탬프 + 화자 + 발언 내용** 형식으로 정리합니다:

```markdown
**MM:SS 화자**
발언 내용을 여기에.
```

예시:

```markdown
**02:54 C**
저도 공유해 보려고 하는데요. 저는 에이전트를 어떻게 잘 사용하는지에 대해서 좀 많이 해봤던 것 같아요.

**03:30 A**
들립니다, 이제.
```

세션이 1시간을 넘어 시·분·초가 필요하면 `HH:MM:SS` 형식도 가능합니다.

### 화자 표기

기본은 **A, B, C, …** 익명 표기입니다. 사용자가 닉네임·실명을 명시한 경우에만 그것을 사용합니다.

이전 회차와 연속되는 세션이라면 **이전 회차의 화자 표기와 반드시 일치**시킵니다. 신규 합류자만 다음 알파벳을 부여합니다.

짧은 추임새("응", "네")만 단발로 등장하는 화자(STT가 잘못 분리한 가능성이 큼)는 별도 인물로 만들지 말고, 가까운 발언자에게 통합하거나 정리 과정에서 제외합니다.

### 음성 인식 오류 보정

다음을 적극적으로 보정하되, **확실한 것만** 보정하고 불확실한 것은 추정 표기 후 주석으로 남깁니다.

- **기술 용어·도구명**: 영문 약어·외래어가 가장 자주 오인식됩니다. 대화 맥락에서 의미가 통하면 정식 명칭으로 교체
- **고유명사·인명**: 회사명·제품명·인명은 맥락으로 추정 가능하면 교체. 인명이 의심스러우면 사용자에게 확인 요청
- **숫자·단위·명령어**: `/usage`처럼 슬래시 명령어, 0.2 같은 임계값, 버전 번호는 본래 형태로 복원
- **외래어 표기**: 가능하면 한국어 외래어 표기로 통일하고 처음 등장 시 영문 병기 (예: `옵시디언(Obsidian)`)

### 문장 다듬기

구어체 특성상 반복·헛말·말 끊김이 많습니다. 의미가 유지되는 선에서 문어체에 가깝게 정돈하되, **원래의 말투와 어조는 보존**합니다. 의역하지 말고 최소한으로 다듬으세요.

발언자가 끝맺지 못한 문장은 괄호 `( )`로 의미를 보완합니다. 예: "이 부분에서 (이슈가) 발생했어요."

조사·종결어미를 자연스럽게 보정합니다 (예: "김혜림가" → "김혜림이").

### 보정 사항 주석

검수본 끝에 다음 형식으로 보정 내역을 요약합니다:

```markdown
---

## 검수 시 주요 보정 사항

- **화자 구분**: <화자 표기 정책과 그 이유>
- **음성 인식 오류 보정 (문맥 기반)**
  - "잘못된 STT" → `정정된 용어` (필요시 부연 설명)
  - …
- **불확실한 부분**: <음성 뭉침으로 확신이 없는 부분과 추정 처리 방식>
- **민감 맥락**: <사적/민감 정보가 있다면 어떻게 처리했는지>
- **문장 다듬기**: <어떤 원칙으로 다듬었는지 한 줄 요약>
```

이 섹션은 사용자가 검수본을 신뢰하고 검증할 수 있게 하는 핵심 요소이므로 빠뜨리지 마세요.

## 요약본 작성 규칙

### 머리말 형식

```markdown
# <세션명> 요약 (대략 <소요 시간>)

## 개요
- **주제**: <한 줄 요약>
- **참석**: <화자 목록과 각자의 역할/특이사항>
- **방식**: <세션 진행 방식>
```

### 본문 구조

요약본은 **시간 순 받아쓰기가 아니라 토픽별 재구성**입니다. 대화에서 다룬 주요 논의를 토픽으로 묶어 섹션을 구성합니다.

전형적인 섹션 구성:

1. **발표/공유 내용** - 각자가 가져온 결과물/계획을 화자별 또는 주제별로 정리
2. **핵심 토론** - 여러 화자가 의견을 주고받은 주요 논점. 화자별 입장을 명확히 구분
3. **부수 논의** - 사이드 화제, 도구·환경 잡담 등
4. **다음 액션 (To-Do)** - 표 형식으로 정리 권장
5. **키워드** - 끝에 검색용 키워드 목록

섹션 제목은 내용에 맞춰 유연하게 작성합니다. 정형 템플릿을 강요하지 마세요.

### 사용자 메모 반영

사용자가 회의 메모를 함께 제공했다면, 메모에 담긴 회고 포인트·결정 사항을 별도 섹션(예: "발표자 메모에 담긴 추가 인사이트")으로 만들거나 관련 토픽에 자연스럽게 흡수시킵니다. 메모는 녹음에 드러나지 않은 사후 회고를 담고 있는 경우가 많으므로 빠뜨리지 마세요.

### 요약 분량

A4 1~3페이지 분량이 적절합니다. 길어도 4페이지를 넘기지 마세요. 받아쓰기가 아니라 **재구성된 핵심**이어야 합니다.

### 화자별 입장 보존

토론 섹션에서는 어떤 화자가 어떤 주장을 했는지 명확히 표기합니다. 예:
- "A는 X를 제안 - 이유는 …"
- "B의 반론 - …"
- "공통 합의 - …"

## 통합본 작성 규칙

같은 시리즈의 회차가 누적되어 사용자가 "전·후반부 합쳐줘", "1·2회차 통합본 만들어줘"라고 요청할 때 적용합니다.

### 통합 검수본

각 회차의 본문(발언 구간)만 추출해 순서대로 이어 붙입니다. 머리말과 보정 사항 섹션은 통합용으로 새로 작성합니다.

**타임스탬프 구분**: 회차가 섞이지 않도록 각 타임스탬프 앞에 회차 표시를 붙입니다. 예: `[전] 02:54 C`, `[후] 00:01 A` 또는 `[1회차]`, `[2회차]`.

회차 사이에는 명확한 구분 헤더를 둡니다:

```markdown
## 1부 - 전반부

<전반부 본문>

---

## 2부 - 후반부

<후반부 본문>
```

### 통합 요약본

회차별 요약을 단순 이어붙이는 게 아니라, **회차를 가로질러 같은 주제를 묶어 재구성**합니다. 한 발표자가 여러 회차에 걸쳐 같은 프로젝트를 발전시킨 경우, 그 발전 과정을 하나의 섹션으로 통합합니다.

회차에 걸친 핵심 토론은 별도 섹션으로 묶어 명확히 보여줍니다.

## 예시 (참고용 단편)

### 검수본 발언 예시

원본 STT:
```
02:54 화자 3
네. 저도 공유해 보려고 하는데요. 저는 에이전트를 어떻게 잘 사용하는지에 대해서 좀 많이 해봤던 것 같아요. 실제로 스킨스 같은 것도 만들어 보고
```

검수본:
```markdown
**02:54 C**
저도 공유해 보려고 하는데요. 저는 에이전트를 어떻게 잘 사용하는지에 대해서 좀 많이 해봤던 것 같아요. 실제로 스킬 같은 것도 만들어 보고요.
```

보정: "스킨스" → `스킬`, 종결어미 다듬기, 노이즈성 "네." 제거.

### 보정 사항 항목 예시

```markdown
- **음성 인식 오류 보정 (문맥 기반)**
  - "오마이 클로드 코드" → `Oh My Claude Code`
  - "딥 인터뷰 / 디빙터비 / 디벤트" → `딥 인터뷰(Deep Interview)`
  - "스톡처드 아웃풋" → `structured output(구조화된 출력)`
```

### 요약본 토픽 섹션 예시

```markdown
## 3. 핵심 토론 - 해커톤에서의 AI 협업

### C의 고민
- 소크라틱 런 해커톤에서 리드를 맡아, 플랜을 짜서 팀원들에게 분배하려 했으나 잘 안 됨
- 클로드 코드로 플랜 작성 + 코덱스로 구현. **딥 인터뷰 후 컨센서스 플랜까지 돌린 게 패착**

### D의 제안 - 에픽/태스크 구조
- 회사식 **에픽(Epic) → 태스크** 순서로 접근 권장
- …
```

## 주의 사항

- **불확실한 부분은 추정 표기**: 음성 뭉침으로 확신이 없는 단어는 "(추정)" 표시를 달거나, 보정 사항 섹션에 명시합니다. **잘못된 추측을 확정으로 적지 마세요.**
- **사용자 메모와 충돌**: STT와 사용자 메모가 충돌하면 사용자 메모를 우선합니다. 메모는 발표자가 직접 적은 것이므로 정확도가 높습니다.
