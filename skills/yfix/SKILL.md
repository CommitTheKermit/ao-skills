# yfix

버그나 문제를 **곧장 고치지 않고**, 먼저 (1) 왜 그렇게 동작하는지 근본 원인을 코드 근거와 함께 설명하고, (2) 해결책 후보들을 트레이드오프로 비교한 뒤, (3) 사용자에게 어느 방향으로 갈지 선택을 묻는다. 실제 코드 수정은 사용자가 방향을 고른 뒤 **이 스킬 밖의 일반 대화 흐름**에서 진행한다.

## 책임 경계 (반드시 지킬 것)

- 이 스킬의 산출물은 **원인 설명 + 트레이드오프 비교 + 선택 질문**까지다.
- 분석 단계에서 코드를 **읽기만** 한다. 절대 이 스킬 안에서 파일을 수정/생성하지 않는다.
- 사용자가 선택지를 고르면, 그 선택을 확인한 뒤 일반 작업 흐름으로 넘어가 수정한다(그 수정은 yfix의 산출물이 아니다).

## 발동

- 명시적 호출: `/yfix [증상/문제 설명]`
- 자동 감지: 사용자가 버그/문제 수정 맥락("왜 안 돼", "이거 고쳐줘", "왜 이렇게 동작해" 등)을 드러내면 발동.
- **자명한 것은 건너뛰기**: 오타, 단일 줄 명백한 실수처럼 원인과 해법이 모두 명백하면 절차를 생략하고 바로 고쳐도 된다. 원인이 애매하거나 해법이 갈릴 때만 이 절차를 밟는다.

## 절차

### 1. 근본 원인 조사 (코드 실제 조사)

- 증상과 관련된 코드를 직접 읽어 **왜 이렇게 동작하는지** 메커니즘을 규명한다.
- **근본 원인 vs 증상**을 구분한다. 눈에 보이는 증상이 아니라, 그 증상을 만들어내는 실제 원인을 짚는다.
- 모든 원인 주장에는 **근거 `파일:라인`**을 붙인다(추측이면 "추정"이라고 명시).
- 원인이 여러 갈래로 가능하면 가설을 나열하고, 코드로 확인 가능한 것은 확인해 좁힌다.

출력 예시:

```
## 원인
`src/cart/total.ts:42` 에서 `parseInt(price)` 가 "1,200" 의 콤마 앞까지만 파싱해
1200 대신 1 이 된다. 합계가 과소 계산되는 근본 원인. (증상: "결제 금액이 이상함")
```

### 2. 해결책 후보 트레이드오프 비교 (설명표)

- 가능한 해결책 후보를 제시한다. **개수는 상황에 따라 유동적**이다 - 의미 있는 대안만큼만 낸다. 해법이 사실상 하나뿐이면 그 하나만 내고 "왜 다른 대안이 없는지" 이유를 적는다.
- 각 후보를 **마크다운 표**로 비교한다. 권장 열:

| 후보 | 접근/원리 | 장점 | 단점·트레이드오프 | 리스크/영향 범위 |
|------|-----------|------|-------------------|------------------|

- 표 아래에 한 줄로 **추천안과 그 이유**를 덧붙인다(강요하지 않음, 판단 보조용).

### 3. 사용자에게 선택 질문 (AskUserQuestion)

- 표로 비교를 보여준 **뒤**, `AskUserQuestion` 도구로 어느 후보로 갈지 묻는다.
- 각 옵션 `label`은 후보 이름, `description`은 그 후보의 핵심 트레이드오프 한 줄.
- 옵션 수는 후보 수에 맞춘다(유동적). 사용자가 "Other"로 직접 방향을 줄 수 있음을 활용한다.
- 후보가 단 하나뿐이면, 선택 대신 "이 방향으로 진행할까요?" 확인 질문으로 바꾼다.

### 4. 선택 후

- 사용자가 고른 방향을 한 줄로 복창하고, **여기서 yfix는 끝**난다.
- 이후 실제 수정은 일반 작업 흐름으로 진행한다(원하면 곧바로 이어서 구현).

## 하지 말 것

- 원인 설명 없이 바로 해결책으로 직행하지 않는다.
- 근거 없이 원인을 단정하지 않는다("아마 ~때문" 식이면 "추정"이라고 표기).
- 후보를 인위적으로 2~3개로 늘리거나, 의미 있는 대안을 1개로 뭉개지 않는다(유동적).
- 사용자가 고르기 전에 코드를 수정하지 않는다.
